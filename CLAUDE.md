## Conventions

1. **TypeScript preferred** – Use TypeScript for all new code where reasonable (frontend always, backend when practical).
2. **REST endpoints** – All API routes live under `/api` prefix.
3. **No secrets committed** – All credentials and secrets read from environment variables (`.env` files are gitignored).
4. **Naming** – Use kebab-case for files/folders, PascalCase for React components, camelCase for JS/TS variables and functions.
5. **Imports** – Prefer absolute imports with `@/` alias in the Next.js frontend.
6. **Error handling** – All API endpoints return consistent error format: `{ error: string, details?: any }`.

## AI Working Rules

> These rules MUST be followed by any AI agent working in this repo.

1. **Always read context first** – Before making large changes, read `spec-template.md`, the relevant spec in `specs/`, and the relevant plan in `plans/`.
2. **Follow the four work phases** – PLAN → IMPLEMENT → REVIEW → VERIFY. See "Work Phases" below.
3. **No new top-level folders** – Do not create folders outside the established layout without explicit permission.
4. **Spec-driven** – Every feature must trace back to a spec document.
5. **Plan-first** – Create or update a plan in `plans/` before implementing substantial changes.
6. **Review-driven** – Stop and request review at major milestones (spec draft, plan, backend complete, frontend complete, CI changes).
7. **Branch discipline** – Work only on the current worktree/branch. Never run `git merge`, `git rebase`, or `git reset --hard` automatically. Leave all merge and branch integration decisions to the human.

## Work Phases

> Every non-trivial change — backend or frontend — MUST pass through all four phases in order.

### Phase 0: RESEARCH (when needed)

> **Applies to:** non-trivial or unfamiliar domains — new AWS services, complex animations, security patterns, libraries not yet used in this repo. **Skip for routine, well-understood work.**

1. Identify what is unfamiliar in the upcoming change.
2. Use the **search-first** skill to research official docs, best practices, and example implementations.
3. Produce a brief **Research Artifact** (`plans/<slug>-research.md`) with options considered, recommended approach, and risks.
4. **⏸️ STOP — Wait for user approval before proceeding to PLAN.**

### Phase 1: PLAN

1. Read the relevant spec in `specs/`, any existing codemap in `docs/CODEMAPS/`, and research findings (if Phase 0 was done).
2. Create or update `plans/<slug>-plan.md` with 10–20 discrete, testable tasks grouped by area.
3. Reference prompts (`prompts/00-product-definition.md`, `prompts/01-schema-design.md`, `prompts/02-api-backend.md`, `prompts/03-frontend-ui.md`) as applicable.
4. **⏸️ STOP — Wait for user approval of the plan before proceeding.**

### Phase 2: IMPLEMENT

1. Execute **only** the tasks listed in the approved plan — no scope creep.
2. Backend work goes in `apps/api-node/`; frontend work goes in `apps/web/`.
3. Write or update tests alongside implementation (not deferred to a later phase).
4. **⏸️ STOP — Wait for user review after each major milestone (backend complete, frontend complete).**

### Phase 3: REVIEW

1. Run the `/code-review` command (or delegate to the code-reviewer agent) against all changed files.
2. Produce a short **Review Artifact** summarizing: issues found, suggestions, risk areas.
3. Fix any critical issues before proceeding; document accepted trade-offs.
4. **⏸️ STOP — Wait for user sign-off on the review artifact.**

### Phase 4: VERIFY

1. Run backend tests: `cd apps/api-node && npm test`.
2. Run frontend build: `cd apps/web && npm run build`.
3. If failures occur, use the `/build-fix` command (build-error-resolver agent) or `/tdd` to diagnose and fix.
4. Repeat until all tests pass and build succeeds.
5. **Do NOT recommend deployment until VERIFY succeeds with zero failures.**
6. **⏸️ STOP — Final review. After approval, commit and push.**

## Quality Bars

> No task is "done" until it meets these minimum bars. Agents MUST self-check before marking work complete.

### Backend Changes

- [ ] Tests added or updated for every new/changed endpoint and service function.
- [ ] Error paths handled — invalid input returns `{ error, details? }` with appropriate HTTP status.
- [ ] Edge cases considered — empty arrays, null values, missing optional fields.
- [ ] Logging in place — structured log entries for errors and significant operations (no sensitive data logged).
- [ ] No hardcoded secrets — all config from environment variables.
- [ ] SQL queries parameterized — no string concatenation.

### Frontend Changes

- [ ] Responsive — tested at mobile (375px), tablet (768px), and desktop (1280px) breakpoints.
- [ ] Accessible basics — semantic HTML, `alt` attributes, keyboard navigation, focus management.
- [ ] `prefers-reduced-motion` respected — Framer Motion animations wrapped in motion-safe checks.
- [ ] Design consistent with tokens — using the established color palette, spacing scale, and typography (Inter).
- [ ] Loading, error, and empty states handled for every data-fetching component.
- [ ] No inline secrets or API keys in client-side code.

### CI/CD Changes

- [ ] Test gate preserved — deploy job still depends on passing tests via `needs: test`.
- [ ] No removal of existing tests without justification and approval.
- [ ] Build step still runs `npm run build` for frontend verification.
- [ ] No new secrets added to CI without explicit approval.

### Pre-Completion Checklist

Before marking any task as complete, run through this mentally:

1. **Tests pass?** — `npm test` in `apps/api-node/` is green.
2. **Build succeeds?** — `npm run build` in `apps/web/` exits cleanly.
3. **Spec covered?** — every user story in the spec is addressed or explicitly deferred.
4. **Codemap current?** — if structural changes were made, codemaps are updated.
5. **No dead code?** — removed unused imports, functions, and files.
6. **No TODO debt?** — any `TODO` comments have corresponding tracked issues or are resolved.
7. **Critical paths hardened?** — for any flow labeled as critical in the spec (auth, payments, data mutations), run the `critical-path-hardening` workflow (`.agent/workflows/critical-path-hardening.md`) before calling the feature done.

### Security & Compliance

- [ ] Secrets scan passes — the CI pipeline runs [Gitleaks](https://github.com/gitleaks/gitleaks) on every push and PR to `main` (`.github/workflows/secrets-scan.yml`). Deploy is blocked if potential secrets are detected.
- [ ] No secrets in code or docs — use environment variables for real credentials. Examples and documentation must use obviously fake placeholders, never real keys.
- [ ] PII handled safely — logs use user IDs or hashed values, never raw PII (emails, phone numbers, addresses).

### Performance

- [ ] For performance-sensitive features (high-traffic endpoints, real-time APIs, data-heavy queries), run the `perf-baseline` workflow (`.agent/workflows/perf-baseline.md`) to establish latency and error-rate baselines before calling the feature done. Test scripts live in `perf/` using [k6](https://k6.io/).

## Analytics

For mature apps — those past initial launch with real users — running the `analytics-insights` workflow (`.agent/workflows/analytics-insights.md`) is strongly encouraged. The workflow reads an app's spec and database schema, identifies 3–5 key product questions (activation, retention, feature adoption, errors), and generates tailored SQL queries and dashboard recommendations. Tying features to measurable metrics ensures that development effort is guided by data, not assumptions. The generic events schema in `analytics/schema-notes.md` and the query templates in `analytics/queries-template.md` provide a starting point that any app can adopt.

## Model Tiers

Use the right model for the right task. Over-using expensive models wastes budget; under-using them risks quality on critical work.

| Tier | Model | When to Use | Examples |
|------|-------|-------------|----------|
| **Tier 1** — Deep | Claude Opus / best available | Research, architecture, security audits, critical-path hardening, complex debugging, governance audits | `research-director`, `security-scan`, `critical-path-hardening`, `ops-playbook`, `audit-workflow` |
| **Tier 2** — Standard | Claude Sonnet | Feature implementation, UI development, code review, workflow execution, refactoring | `new-production-app`, `platform-new-app`, `self-review`, `analytics-insights`, `perf-baseline` |
| **Tier 3** — Fast | Claude Haiku / fast model | Test generation, documentation, small fixes, formatting, simple CRUD, boilerplate | `e2e-test-gen`, `decision-digest`, `build-error-resolver`, doc updates, linting fixes |

### Selection Guidelines

- **Default to Tier 2** for most implementation work.
- **Escalate to Tier 1** when the task involves:
  - Security or compliance decisions
  - Architectural choices that are hard to reverse
  - Debugging production incidents
  - Analyzing unfamiliar codebases or complex state machines
- **Drop to Tier 3** when the task is:
  - Mechanical or template-driven (generating tests from existing patterns)
  - Documentation-only with no design decisions
  - Single-file fixes with clear scope

### Workflow Model Annotations

Each workflow file may include a `model_tier` recommendation in its frontmatter or description. When present, agents should prefer the recommended tier. When absent, default to Tier 2.

## Agent Roles

### 1. Planning Agent

**Responsibility:** Research, spec creation, plan creation, and architecture decisions.

- Reads the app idea and expands it into a full spec using `spec-template.md`.
- Creates implementation plans in `plans/`.
- Produces review artifacts at spec and plan stages.

**Constraints:**
- Does NOT write application code.
- Does NOT modify `infra/`, `.github/`, or deployment configs.
- Must stop for review after producing spec and plan drafts.

---

### 2. Backend Agent

**Responsibility:** Implement API endpoints, database models, and server-side logic.

- Works exclusively in `apps/api-node/`.
- Reads the approved spec and plan before coding.
- Implements Express routes, middleware, and database queries.
- Creates and runs tests for implemented endpoints.

**Constraints:**
- Only modifies files in `apps/api-node/`.
- Does NOT touch frontend code, infra, or CI/CD configs.
- Reads `DATABASE_URL` and other config from environment variables only.
- Must stop for review after completing backend implementation.

---

### 3. Frontend Experience Agent

**Responsibility:** Build high-quality UI/UX using Next.js, Tailwind, and Framer Motion.

- Works exclusively in `apps/web/`.
- Reads the approved spec and plan before coding.
- Implements pages, components, layouts, and API integrations.
- Targets top-tier modern SaaS design: strong typography, visual hierarchy, grid discipline, micro-interactions.
- Ensures responsive design (mobile + desktop).

**Constraints:**
- Only modifies files in `apps/web/`.
- Does NOT touch backend code, infra, or CI/CD configs.
- Must wire API calls to the backend endpoints defined in the spec.
- Must stop for review after completing frontend implementation.

---

### 4. Test & CI Agent

**Responsibility:** Verify quality, run tests, and maintain CI/CD pipelines.

- Runs existing tests and reports results.
- Checks `.github/workflows/deploy-ec2.yml` and `infra/deploy/deploy.sh` for correctness.
- Proposes CI/CD changes as review artifacts before modifying.

**Constraints:**
- Does NOT implement new features.
- Must propose CI/CD changes as a plan and wait for approval before editing pipeline files.
- Must run the full test suite after any changes.

## Codemap & Memory

> Codemaps and learned patterns are the repo's **architectural memory**. Agents MUST consult them before structural changes and keep them current.

### Codemaps

Codemaps are token-lean architecture summaries generated by the `/update-codemaps` command. They live in the repo at:

```
docs/CODEMAPS/
  architecture.md    — High-level system diagram, module boundaries, data flow
  backend.md         — Express routes, services, middleware, DB access layer
  frontend.md        — Next.js pages, components, layouts, client/server split
  data.md            — Database schema, relationships, migration history
  dependencies.md    — External dependencies, versions, integration points
```

Diff reports and analysis are stored in `.reports/codemap-diff.txt`.

**When to generate/update codemaps:**

| Trigger | Action |
|---------|--------|
| After completing a major feature | Run `/update-codemaps` to capture new modules, routes, and components. |
| After a non-trivial refactor | Run `/update-codemaps` — structural changes invalidate existing maps. |
| Before planning large changes | **Read** the relevant codemap(s) first — understand module responsibilities before proposing changes. |
| When introducing a new module | Update the relevant codemap to document the module's responsibility and boundaries. |
| When deprecating/removing a module | Update the relevant codemap to remove stale entries. |

**How to use codemaps before changes:**

1. Read `docs/CODEMAPS/architecture.md` to understand the overall system.
2. Read the domain-specific codemap (e.g., `backend.md` for API changes).
3. Check `dependencies.md` if your change adds or modifies external integrations.
4. Reference the codemap in your plan to justify why a new module is needed (or why an existing one should be extended instead).

### Memory & Continuous Learning

Learned patterns are saved by the `/learn` command (manual) or the Stop hook (automatic) to:

```
~/.claude/skills/learned/    — Extracted patterns, error resolutions, workarounds
```

**When to use:**

| Trigger | Action |
|---------|--------|
| End of a productive session | Run `/learn` to extract reusable patterns before closing. |
| After solving a non-trivial bug | Run `/learn` to save the error-resolution pattern. |
| Starting a new session | The session-start hook auto-loads prior context. |
| Instinct review | Run `/instinct-status` to see learned instincts and confidence scores. |

### Session Reflections

After major missions (e.g., `new-app-from-idea`, large refactors), agents should produce a structured reflection and save it to:

```
docs/reflections/YYYY-MM-DD-<slug>.md
```

Each reflection should cover:
1. **What changed** — summary of files added, modified, or removed.
2. **What worked** — patterns, tools, or decisions that saved time.
3. **What didn't** — pain points, dead-ends, or wasted effort.
4. **Suggested improvements** — prompt tweaks, new rules, or process changes.

**Before large new changes**, read the 2–3 most recent reflections to avoid repeating past mistakes.

### File Location Summary

| Artifact | Location | Tracked in Git? |
|----------|----------|-----------------|
| Codemaps | `docs/CODEMAPS/*.md` | ✅ Yes |
| Decision docs | `docs/decisions/*.md` | ✅ Yes |
| Decision digests | `docs/decisions/weekly-digest-*.md` | ✅ Yes |
| Session reflections | `docs/reflections/*.md` | ✅ Yes |
| Codemap diff reports | `.reports/codemap-diff.txt` | Optional (gitignore-able) |
| Learned skills | `~/.claude/skills/learned/` | No (user-local) |
| Instincts (v2) | `~/.claude/skills/continuous-learning-v2/` | No (user-local) |

Together, codemaps, session reflections, and decision docs form the lab's **learning and traceability system**. Codemaps capture _what_ the code looks like, reflections capture _how_ a session went, and decision docs capture _why_ a choice was made. Agents should consult all three before proposing structural changes, and create decision docs (using `docs/decisions/template.md`) when making choices that constrain future options. The `decision-digest` workflow (`.agent/workflows/decision-digest.md`) can generate weekly summaries for team distribution.

## AI Production Lab

> This repo is an **AI production lab** — a template and toolkit for turning raw ideas into secure, tested, deployable apps as fast as possible.

### Stack

**Frontend:** Next.js + React + Tailwind CSS + Framer Motion · **Backend:** Node/Express + PostgreSQL + Amazon S3 · **Infra:** AWS EC2 + GitHub Actions CI/CD.

### Agents & Workflows

| Agent / Workflow | Role |
|------------------|------|
| `new-app-from-idea` | **App factory** — takes a product idea through spec → plan → implement → verify in one orchestrated run. |

### Production Principles

- **Spec-driven and plan-first.** Every feature traces to a spec; every implementation starts with an approved plan.
- **Tests and CI must pass before deploy.** No code ships until `npm test` and `npm run build` are green.
- **Security and basic observability checks before "done."** Security scans, structured logging, and error handling are mandatory — not optional polish.
- **No new top-level folders or infra changes without human approval.** Structural and infrastructure changes require explicit sign-off to prevent drift.

## Self-Improvement

Beyond polishing recently-completed features (`self-review`), agents can proactively scan and upgrade the **entire codebase** using the self-improvement system.

- **`self-improvement/strategy.md`** — defines improvement categories (Code Quality, Performance, Security, DX, Architecture), agent constraints, and the report format.
- **`self-improvement/checklist.md`** — item-level evaluation criteria agents use when scanning code for upgrade opportunities.
- **`.agent/workflows/self-upgrade.md`** — the agent-invocable workflow. Run with a `scope` (target directory) to: scan → produce a prioritized report → wait for approval → apply fixes one-by-one with test re-runs.

**Key constraints:**
- Human approval required before any changes are applied.
- Tests must stay green — any change that breaks tests is reverted.
- Max 10 items per run, max 2 retries per item.
- Architecture findings require a Tier 1 model.

**Relationship to other workflows:**

| Workflow | Scope | Trigger |
|----------|-------|---------|
| `self-review` | Recently-completed feature files | After feature completion |
| `self-upgrade` | Any part of the codebase | Manual or scheduled |
| `security-scan` | Security-specific deep audit | Before release |
| `perf-baseline` | Performance benchmarking | Performance-sensitive features |

The `platform-new-app` workflow (`.agent/workflows/platform-new-app.md`) integrates available platform modules when scaffolding new apps.

### App Registry (`labs-config.yaml`)

Central registry of all apps, read by agents, n8n workflows, and Amazon Q. Each entry records `name`, `owner`, `envs`, `sla` tier, `last_deploy`, `critical_paths`, and `tags`. See `docs/labs-scaling.md` for full details.

### Governance (`governance/`)

Enterprise policies and compliance automation:

- **`governance/policy.yaml`** — SLA tier definitions, security baselines, compliance requirements, audit schedules.
- **`.agent/workflows/governance-audit.md`** — agent-invocable quarterly audit across all registered apps.
- **`governance/n8n-schedule.md`** — consolidated cron schedule for all automated governance workflows.

### Agent Rules for Multi-App Work

1. **Read `labs-config.yaml`** before any cross-app operation.
2. **Respect SLA tiers** from `governance/policy.yaml` — apply appropriate quality gates per tier.
3. **Reference `platform/` modules** when shared functionality is needed — do not duplicate.
4. **Notify owners** — use the `owner` field to route audit findings and change notifications.
